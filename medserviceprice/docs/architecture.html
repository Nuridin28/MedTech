Event-drivenИнгестия асинхронна: API/Beat кладут задачу в Redis-очередь, Celery Worker её разбирает. Развязка по очереди сообщенийДа — слой сбора

Итог: гибрид «модульный монолит + event-driven ингестия» поверх клиент-серверной модели. Чтения идут синхронно (API → PostgreSQL / Redis), а тяжёлый сбор данных вынесен в асинхронный конвейер на Celery — это даёт изоляцию источников и не блокирует API.

5 Анализ архитектуры

Узкие места, масштабирование, безопасность, отказоустойчивость, направления улучшений.

Узкие места

  • Единственный PostgreSQL. Поиск (trgm/vector), запись парсера и чтения API идут в одну БД без реплик — под нагрузкой чтения конкурируют с ночной записью.
  • Worker concurrency = 2. При росте числа источников парсинг удлиняется, задачи сериализуются.
  • attach_offer_stats. Делает отдельный запрос на каждый из 8 хитов автокомплита (N+1) — смягчено кэшем, но остаётся точкой роста.
  • Семантический поиск. Загружает многоязычный трансформер в память worker — высокий расход RAM при включении.

Масштабирование

  • Backend API — stateless, горизонтально масштабируется за балансировщиком тривиально (состояние в PostgreSQL/Redis).
  • Worker'ы масштабируются добавлением реплик (Celery разбирает общую очередь). Источники уже изолированы: один источник = одна задача.
  • PostgreSQL — вертикально + read-replica для чтений; вынести vector-поиск на реплику.
  • Redis — при росте нагрузки разнести broker / cache / rate-limit на разные инстансы или Redis Cluster.

Безопасность

  • естьrate-limit 120/мин (Redis), CORS на конкретный origin, admin за X-API-Key, security-заголовки (приложение + Nginx), только ORM (без склейки SQL), page_size ≤ 100, валидация Pydantic, глобальный обработчик исключений (стектрейс не утекает).
  • естьхранятся только публичные цены, без PII пациентов; парсер уважает robots.txt и задержку.
  • пробелнет аутентификации пользователей (аккаунт = localStorage); admin-ключ статичный (нет ротации/лимита на сам ключ); TLS терминируется вне compose; нет аудита admin-действий.

Отказоустойчивость

  • естьпартиал-толерантность: ошибка строки → partial, ошибка источника → failed, но другие источники не падают (изоляция задач).
  • естьretry с экспонентой в парсере; healthchecks + restart: unless-stopped у всех контейнеров; инвалидация кэша обёрнута в try/except и не валит задачу.
  • рискSPOF: один инстанс PostgreSQL и Redis (нет реплик/бэкапов в compose), один host; Beat — единственная точка планирования.

Потенциальные улучшения

  • Миграция на оркестратор (Kubernetes / Swarm) + load-balancer + терминация TLS; автоматические бэкапы PostgreSQL (pgBackRest / WAL-G).
  • Реализовать отправку price-alerts (sender для subscriptions) — сейчас данные копятся, рассылки нет.
  • Подключить реальный источник курса вместо fallback 470; свести attach_offer_stats к одному агрегирующему запросу.
  • Observability: структурированные логи → Loki, метрики Celery/FastAPI → Prometheus/Grafana, трейсинг конвейера парсинга.
  • Включить и прогреть семантический поиск на отдельном worker-пуле; вынести embeddings в отдельный сервис.
  • Backend-аутентификация (JWT/сессии) для аккаунт-функций вместо localStorage, если нужны кросс-девайс данные.
