MedServicePrice.kz — Детальная компонентная схема

Полная карта компонентов и ключевых потоков работающей системы: парсинг с дедупликацией, нормализация, потоки запросов к LLM (ассистент + AI-нормализация каталога), и наблюдаемость (структурные логи → ELK, parse_logs, алёрты). Строгий чёрно-белый стиль для печати и ревью.

Источники: KDL · Invitro · doq · файловый импорт  |  модули: api · services · parsers · tasks  |  данные: Postgres · Redis · Elasticsearch

2 Парсинг и дедупликация — пошагово

Конвейер tasks/parsing.py → services/ingest.py. Две независимые дедупликации (сырьё и оффер) + алёрты.

flowchart TB
  start["beat (ежедневно) ИЛИ админ /parse/run"] --> task["parse_source(source_key)"]
  task --> getp["registry.get_parser(key)"]
  getp --> fetch["BaseParser.fetch()
robots.txt · UA · delay 1.5c · retry×3"]
  fetch --> parse["parse() → RawServiceRecord[]
(BeautifulSoup / JSON / файл)"]
  parse --> loop{по каждой записи}

  loop --> h1["content_hash =
sha256(clinic|name|price|currency)"]
  h1 --> d1{"есть в
raw_records?"}
  d1 -->|да| skip1["пропустить сырьё"]
  d1 -->|нет| ins1["INSERT raw_records
(аудит ≥90 дней)"]

  skip1 --> clinic["clinic upsert
(rating/coords/booking refresh)"]
  ins1 --> clinic
  clinic --> nm["normalize(name)"]
  nm --> nmd{"exact ИЛИ fuzzy:
token_set≥88 И token_sort≥60?"}
  nmd -->|да| sid["service_id привязан"]
  nmd -->|нет| uq["→ unmatched_queue (pending)"]

  sid --> h2["offer_hash =
sha256(clinic+name+price)"]
  uq --> h2
  h2 --> d2{"есть offer
(clinic, offer_hash)?"}
  d2 -->|нет| ins2["INSERT service_offers

  d2 -->|да| upd2["UPDATE price/parsed_at/

  upd2 --> ph
  ph -->|да| insph["APPEND price_history"]
  ph -->|нет| loop
  insph --> loop

  loop -->|готово| commit["commit"]
  commit --> log["ParseLog(status, records_count)"]
  log --> alert["evaluate_parse_outcome()
failed → alert · 0 записей → alert"]
  alert --> inval["invalidate cache

Две хэш-дедупликации намеренно разные: content_hash (включает currency) защищает сырьё в raw_records; offer_hash (clinic+name+price) защищает рабочий оффер. История цен пишется только при изменении цены — так строится реальный таймлайн без дублей.

3 Потоки запросов к LLM

Два независимых LLM-пути: публичный ассистент (за safety-gateway) и админская AI-нормализация каталога.

3a. Ассистент — safety gateway (публичный чат)

flowchart LR
  u["Пользователь
вопрос"] --> ep["/api/assistant/chat
rate-limit (мин/день/глоб)"]
  ep --> inj{"injection-паттерны?"}
  inj -->|да| block1["отказ: blocked_input"]
  inj -->|нет| mod{"OpenAI Moderation
flagged?"}
  mod -->|да| block1
  mod -->|нет| scope{"классификатор LLM:
в теме сервиса?"}
  scope -->|нет| off["отказ: refused_offtopic"]
  scope -->|да| ans["LLM с инструментами
(узкий system-prompt)"]
  ans --> tc{"модель вызвала
инструмент?"}
  tc -->|"да (до 3 раундов)"| tool["search_services /
cheapest_clinics / checkup_basket
→ живая БД"]
  tool --> ans
  tc -->|нет| leak{"утечка canary /
системного промпта?"}
  leak -->|да| safe["safe_fallback: blocked_output"]
  leak -->|нет| ok["ответ: answered
(реальные цены)"]

  classDef ctrl fill:#fff,stroke:#000,color:#000,stroke-width:2px;
  classDef dec fill:#cfcfcf,stroke:#000,color:#000,stroke-width:1.5px;
  classDef svc fill:#e9e9e9,stroke:#000,color:#000,stroke-width:1.5px;
  classDef data fill:#f5f5f5,stroke:#000,color:#000,stroke-width:2px;

Ключ OpenAI живёт только на backend. Каждый вызов: rate-limit → injection → модерация → классификатор «в теме?» → генерация с tool-calling (модель вызывает инструменты и берёт реальные цены из БД, не RAG — данные структурные) → проверка утечки. Off-topic и утечки отсекаются.

3b. AI-нормализация каталога (админ, human-in-the-loop)

flowchart LR
  a["Админ
«Сгенерировать (AI)»"] --> q1["/admin/normalization/ai-suggest"]
  q1 -.->|enqueue| task["task: ai_suggest_catalog"]
  task --> load["взять топ unmatched по частоте
+ текущий каталог"]
  load --> llm["LLM: кластеризовать в
name_norm + category + synonyms"]
  llm --> val["валидация:
категория ∈ 4 · синонимы ⊂ списка"]
  val --> store["INSERT catalog_suggestions
(status=pending)"]
  store --> review["Админ: Принять / Отклонить"]
  review -->|принять| apply["создать ServiceCatalog +
привязать офферы синонимов +
закрыть unmatched"]
  review -->|отклонить| rej["status=rejected"]
  apply --> renorm["task: renormalize_offers
(перепрогон матчера)"]
  renorm --> cov["покрытие ↑"]

Инвариант: LLM только предлагает — каталог расширяется лишь после подтверждения аналитиком. При подтверждении синонимы сразу привязывают свои цены, затем перенормализация дотягивает остальное.
